## 1. Application Startup Flow

### What happens when the `.exe` is double-clicked?

```
Double-click RalseiWarehouse.exe
  │
  └─► App.OnStartup()                          [App.xaml.cs:18]
       │
       ├─► base.OnStartup(e)                   // WPF internal init (loads App.xaml resources)
       │
       ├─► Load appsettings.json               // Read JSON file from the .exe directory
       │    ├─ new ConfigurationBuilder()
       │    ├─ .SetBasePath(AppContext.BaseDirectory)  // Folder where .exe lives
       │    ├─ .AddJsonFile("appsettings.json")         // Read the file
       │    ├─ .Build()                                 // Build IConfiguration object
       │    └─ .GetConnectionString("RalseiWarehouse")  // Extract "ConnectionStrings.RalseiWarehouse"
       │         │
       │         └─ If missing → CRASH: "The database connection string is missing."
       │
       ├─► Build DI Container (ServiceCollection)
       │    │
       │    ├─► AddDbContextFactory<WarehouseDbContext>()
       │    │    └─ Configures EF Core to use SQL Server with the connection string.
       │    │       This registers a FACTORY, not a context instance.
       │    │       Every operation later calls factory.CreateDbContextAsync() for a fresh context.
       │    │
       │    ├─► AddSingleton<IWarehouseService, WarehouseService>()
       │    │    └─ ONE instance for the entire app lifetime.
       │    │       It holds `currentUser` — the logged-in session.
       │    │       If it were transient, each ViewModel would get a different instance
       │    │       and login would be lost.
       │    │
       │    └─► AddTransient for all 5 ViewModels
       │         └─ Each time you ask for one, you get a brand-new instance.
       │            LoginViewModel, MasterDataViewModel, TransactionViewModel,
       │            InventoryViewModel, AdministrationViewModel
       │
       ├─► services.BuildServiceProvider()
       │    └─ Freezes the container. No more registrations allowed.
       │
       └─► ShowLogin()                          [App.xaml.cs:28]
```

### ShowLogin() — the login-to-main transition

```
ShowLogin()
  │
  ├─► GetRequiredService<LoginViewModel>()      // DI creates a fresh LoginViewModel
  │    └─ LoginViewModel's constructor receives IWarehouseService (the singleton)
  │
  ├─► Create LoginWindow
  │    └─ login.DataContext = vm                // All XAML bindings now route to LoginViewModel
  │    └─ MainWindow = login                    // App shuts down when THIS window closes
  │
  ├─► Subscribe to vm.Succeeded event
  │    └─ The lambda (user => { ... }) will fire when login passes.
  │       This is the ONLY place that knows how to switch windows.
  │       The LoginViewModel just fires an event — it doesn't know what happens next.
  │
  └─► login.Show()
       └─ LoginWindow appears. Waiting for user input.
```

### OnExit — cleanup

```
User closes MainWindow
  │
  └─► App.OnExit()
       └─► provider?.Dispose()
            └─ Cascade-disposes all IDisposable services. DbContext factory closes
               connection pools. Singleton service releases resources.
```

### Inside LoginViewModel.LoginAsync()

```
LoginAsync()
  │
  └─► ExecuteAsync( ... )                       [ViewModelBase.cs:13]
       │
       ├─► if (IsBusy) return;    ◄── Guard: ignore double-clicks
       │
       ├─► IsBusy = true           ◄── Locks the button (would disable if bound)
       ├─► Message = string.Empty  ◄── Clear previous errors
       │
       ├─► await operation()       ◄── Run the actual login logic:
       │    │
       │    ├─► if (string.IsNullOrWhiteSpace(UserName) ||
       │    │     string.IsNullOrEmpty(Password))
       │    │     throw new InvalidOperationException("Enter username and password.")
       │    │     └─ This exception is caught by the catch block below
       │    │
       │    └─► var user = await service.LoginAsync(UserName.Trim(), Password)
       │         │  Uppercase L in LoginAsync — this calls the SERVICE, not itself
       │         │
       │         └─► WarehouseService.LoginAsync()    [WarehouseService.cs:75]
       │              │
       │              ├─► factory.CreateDbContextAsync()
       │              │    └─ EF Core opens a new connection to SQL Server
       │              │
       │              ├─► db.Users.Include(x => x.Role)
       │              │          .SingleOrDefaultAsync(x => x.UserName == userName)
       │              │    └─ SQL: SELECT * FROM [User] u
       │              │             INNER JOIN [Role] r ON u.RoleId = r.RoleId
       │              │             WHERE u.UserName = @userName
       │              │
       │              ├─► user is null → return null
       │              │
       │              ├─► Check: does stored password start with "PBKDF2-SHA256$"?
       │              │    │
       │              │    ├─ YES → SecurityHelper.VerifyPassword(password, user.Password)
       │              │    │   │
       │              │    │   ├─► Parse stored string: algorithm$iterations$salt$hash
       │              │    │   ├─► Decode salt and expected hash from Base64
       │              │    │   ├─► Re-compute: Pbkdf2(password, salt, 210000, SHA256, 32)
       │              │    │   └─► FixedTimeEquals(actual, expected)
       │              │    │        └─ Timing-safe comparison (prevents timing attacks)
       │              │    │
       │              │    └─ NO (plain text legacy password) →
       │              │       direct comparison: user.Password == password
       │              │
       │              ├─► If verification fails → return null
       │              │
       │              ├─► If plain text password passed:
       │              │    user.Password = SecurityHelper.HashPassword(password)
       │              │    await db.SaveChangesAsync()
       │              │    └─ AUTO-UPGRADE: Plain text is replaced with PBKDF2 hash
       │              │       Next login uses the hash path. User never knows.
       │              │
       │              ├─► currentUser = user   ◄── Store in singleton for all future checks
       │              └─► return user
       │
       ├─► Back in LoginViewModel:
       │    user is null → throw new InvalidOperationException("Invalid username or password.")
       │    └─ Caught by catch block → Message = "Invalid username or password."
       │
       │    user is NOT null:
       │    └─► Succeeded?.Invoke(user)   ◄── Fire the event!
       │
       ├─► catch (Exception exception)
       │    └─ Message = exception.Message   ◄── Error shown in red TextBlock
       │
       └─► finally { IsBusy = false; }       ◄── Unlock button
```

### Succeeded event fires — window transition

```
Succeeded event fires with authenticated User object
  │
  └─► Lambda in App.ShowLogin() executes:    [App.xaml.cs:31]
       │
       ├─► Resolve all feature ViewModels from DI:
       │    ├─ services.GetRequiredService<MasterDataViewModel>()
       │    ├─ services.GetRequiredService<TransactionViewModel>()
       │    ├─ services.GetRequiredService<InventoryViewModel>()
       │    └─ services.GetRequiredService<AdministrationViewModel>()
       │
       ├─► Create MainViewModel(user, masterData, transactions, inventory, administration)
       │    │
       │    └─► MainViewModel constructor:     [MainViewModel.cs:10]
       │         ├─ Store user, all child VMs
       │         ├─ Parse user.Role.DisplayName
       │         ├─ IsAdministrator = role contains "Admin"
       │         └─ IsManager = IsAdministrator || role contains "Manager"
       │
       ├─► Create MainWindow with DataContext = shell
       ├─► MainWindow = window   ◄── Now the app stays alive for MainWindow
       ├─► window.Show()         ◄── MainWindow appears
       ├─► login.Close()         ◄── LoginWindow closes
       │
       └─► Pre-load data based on role:
            ├─ if (shell.IsManager):   shell.MasterData.LoadAsync()   (fire-and-forget)
            │                          shell.Inventory.LoadAsync()    (fire-and-forget)
            ├─ shell.Transactions.LoadAsync()  (always, fire-and-forget)
            └─ if (shell.IsAdministrator): shell.Administration.LoadAsync()  (fire-and-forget)
```

These LoadAsync calls use `_ =` (discard) — they are fire-and-forget. The data loads in the background while the user sees the main window immediately.

## 3. Role-Based UI Assembly

### How MainWindow xaml decides what tabs to show

```
MainWindow.xaml binds to MainViewModel    [MainWindow.xaml:27-38]

For EACH TabItem, WPF evaluates the Visibility binding:

Tab "Master Data":
  Visibility="{Binding IsManager, Converter={StaticResource BoolToVisibility}}"
  │
  ├─ MainViewModel.IsManager = true  → BoolToVisibility → Visible   → shown
  └─ MainViewModel.IsManager = false → BoolToVisibility → Collapsed → hidden

Tab "Transactions":
  No Visibility binding → ALWAYS Visible (even non-managers see it)

Tab "Inventory":
  Same as Master Data — only Managers/Admins

Tab "Administration":
  Visibility="{Binding IsAdministrator, ...}"
  └─ Only Admins see this tab
```

### How each tab gets its ViewModel

```
For each visible TabItem, the inner UserControl gets its DataContext set:

<TabItem Header="Master Data">
  <views:MasterDataView DataContext="{Binding MasterData}" />
</TabItem>
  │
  └─ Binds to MainViewModel.MasterData (which is MasterDataViewModel instance)

<TabItem Header="Transactions">
  <views:TransactionView DataContext="{Binding Transactions}" />
</TabItem>
  │
  └─ Binds to MainViewModel.Transactions (TransactionViewModel instance)
```

Each ViewModel gets injected with the same IWarehouseService singleton, so they all share the user session.

```
         │         │
         │         ├─► db.Entry(item).State =

         │       (e.g., you saved a supplier that a product references).
         │       This guarantees the UI never shows stale data.
         │
```

```
DeleteSupplierAsync(int id)
  │
  ├─► EnsureManager()     ◄── Only managers/admins can delete
  │
  └─► DeleteAsync<Supplier>(id, "The supplier is used by a product.")
```

**Product ComboBox data context trick:**
```
Each DataGrid row's DataContext = an InputLineViewModel

InputLineViewModel has NO Products property.

So how does the ComboBox get the product list?

The XAML uses RelativeSource:           [TransactionView.xaml:39]
  ItemsSource="{Binding DataContext.Products,
      RelativeSource={RelativeSource AncestorType=DataGrid}}"

Translation:
  1. Walk UP the visual tree until you find a DataGrid
  2. Take that DataGrid's DataContext (which IS TransactionViewModel)
  3. Bind to its .Products property

Result: EVERY row's ComboBox sees the SAME product list,
        but SelectedItem binds to the ROW's own Product property.
```

**Inside WarehouseService.CreateInputAsync():**
```
CreateInputAsync(IEnumerable<InputRequest> lines)
  │
  ├─► EnsureAuthenticated()
  │    └─ Anyone can create imports (no manager/admin check needed)
  │

  │
  ├─► ValidateLines(values.Select(x => (x.ProductId, x.Count)))
  │    │
  │    ├─► if (lines.Count == 0) → throw "Add at least one product with a positive quantity."
  │    ├─► if any ProductId is blank or Count ≤ 0 → throw same error
  │    └─► if any ProductId appears TWICE → throw "A product may appear only once per receipt."
  │
  ├─► if any price < 0 → throw "Prices cannot be negative."
  │
  ├─► factory.CreateDbContextAsync()
  │
  ├─► Create IMPORT HEADER:
  │    var header = new Input
  │    {
  │      InputId = NewId("IN"),          ◄── e.g. "IN-20260720143022-a1b2c3d4e5f6..."
  │      DateInput = DateTime.Now        ◄── Current date/time
  │    }
  │
  ├─► Create LINE ITEMS (one per InputRequest):
  │    header.InputInfos = values.Select(x => new InputInfo
  │    {
  │      Id          = Guid.NewGuid().ToString("N"),  ◄── 32-char unique ID
  │      ObjectId    = x.ProductId,       ◄── Which product
  │      Count       = x.Count,           ◄── How many
  │      InputPrice  = x.InputPrice,      ◄── Purchase cost
  │      OutputPrice = x.OutputPrice,     ◄── Planned sell price
  │      Status      = "Completed"
  │    }).ToList()
  │
  ├─► db.Inputs.Add(header)
  │    └─ EF detects the InputInfos collection → marks those as Added too
  │
  └─► await db.SaveChangesAsync()
       └─ SQL TRANSACTION (auto-wrapped by EF):
            INSERT INTO Input  (InputId, DateInput) VALUES (...)
            INSERT INTO InputInfo (Id, ObjectId, InputId, Count, ...) VALUES (...)
            INSERT INTO InputInfo (Id, ObjectId, InputId, Count, ...) VALUES (...)
            ... (one INSERT per line)
            COMMIT   ◄── All succeed or all roll back together

Back in ViewModel:
         // STEP 2: Clear the input lines (fresh slate)
         InputLines.Clear()
         │
         │  WPF DataGrid sees collection empty → all rows disappear
         │
         // STEP 3: Reload history (shows the new import in History tab)
         Replace(RecentImports, await service.GetRecentImportsAsync())
         │
         │  service returns last 50 imports including the one just created
         │  │
         │  └─► GetRecentImportsAsync()                    [WarehouseService.cs:143]
         │       │
         │       └─► db.Inputs
         │            .Include(x => x.InputInfos)
```

**Inside WarehouseService.CreateOutputAsync() — THE CRITICAL STOCK CHECK:**

```
CreateOutputAsync(int customerId, IEnumerable<OutputRequest> lines)
  │
  ├─► EnsureAuthenticated()
  │
  ├─► if (customerId < 1) → throw "Select a customer."
  │
  ├─► ValidateLines(...)   ◄── Same as import: at least 1 line, positive qtys, no duplicates
  │
  ├─► factory.CreateDbContextAsync()
  │
  ├─► BEGIN TRANSACTION (IsolationLevel.Serializable)  ◄── THIS IS CRITICAL
  │    │
  │    │  Serializable = highest isolation level.
  │    │  It uses RANGE LOCKS on the database:
  │    │    - If another user is ALSO doing an export at the same time,
  │    │      SQL Server makes one wait until the other commits.
  │    │    - This prevents the "double-sell" race condition:
  │    │
  │    │      WITHOUT Serializable:
  │    │        User A reads stock: 10 available
  │    │        User B reads stock: 10 available (before A commits)
  │    │        User A exports 8 → stock becomes 2
  │    │        User B exports 8 → stock becomes -6 (CORRUPTED!)
  │    │
  │    │      WITH Serializable:
  │    │        User A reads stock: 10, locks the rows
  │    │        User B tries to read → WAITS for A
  │    │        User A exports 8, commits, releases locks
  │    │        User B now reads: 2 available
  │    │        User B tries to export 8 → "Insufficient stock. Available: 2."
  │    │
  │    └─ This is the warehouse's most important data integrity guard.
  │
  ├─► FOR EACH LINE in the export:
  │    │
  │    ├─► input = db.InputInfos
  │    │            .Where(x => x.ObjectId == line.ProductId)
  │    │            .SumAsync(x => (int?)x.Count) ?? 0
  │    │    └─ SQL: SELECT SUM(Count) FROM InputInfo WHERE ObjectId = @productId
  │    │              If no rows → NULL → ?? 0 → 0
  │    │
  │    ├─► output = db.OutputInfos
  │    │             .Where(x => x.ObjectId == line.ProductId)
  │    │             .SumAsync(x => (int?)x.Count) ?? 0
  │    │    └─ SQL: SELECT SUM(Count) FROM OutputInfo WHERE ObjectId = @productId
  │    │
  │    ├─► available = input - output    ◄── CURRENT stock (derived, not stored)
  │    │
  │    └─► if (line.Count > available)
  │         throw new InvalidOperationException(
  │           $"Insufficient stock for {line.ProductId}. Available: {available}."
  │         )
  │         └─ If this throws, the Serializable transaction will roll back.
  │            The user sees: "Insufficient stock for SKU-001. Available: 5."
  │
  ├─► All lines passed stock check. Now create the EXPORT:
  │
  │    var header = new Output
  │    {
  │      OutputId = NewId("OUT"),          ◄── e.g. "OUT-20260720143022-..."
  │      DateOutput = DateTime.Now
  │    }
  │
  │    header.OutputInfos = values.Select(x => new OutputInfo
  │    {
  │      Id         = Guid.NewGuid().ToString("N"),
  │      ObjectId   = x.ProductId,
  │      CustomerId = customerId,           ◄── WHO received this export
  │      Count      = x.Count,
  │      Status     = "Completed"
  │    }).ToList()
  │
  ├─► db.Outputs.Add(header)
  ├─► await db.SaveChangesAsync()         ◄── INSERTs within the transaction
  │
  └─► await transaction.CommitAsync()     ◄── RELEASE all locks
       └─ Other waiting transactions can now proceed with updated stock

Back in ViewModel:
         OutputLines.Clear()
         Replace(RecentExports, await service.GetRecentExportsAsync())
         Message = "Export saved. See History tab."
       })
```

### 6.5 Why stock is DERIVED, not stored

```
The inventory is computed as:
  Stock = SUM(InputInfo.Count WHERE ObjectId = X) - SUM(OutputInfo.Count WHERE ObjectId = X)

This is NEVER stored in the Object table. Why?

  1. IMMUTABLE AUDIT TRAIL: Every stock change is a transaction record.
     You can NEVER have stock that doesn't match transaction history.

  2. NO RACE CONDITIONS on a stock column:
     If stock were a column, two concurrent exports would both do:
       Read stock (10) → Subtract 8 → Write stock (2)
       Read stock (10) → Subtract 8 → Write stock (2)    ← WRONG, should be -6
     Serializable transaction prevents this.

  3. FULL HISTORY: You can reconstruct stock at any point in time.

  The cost: stock is computed on every read (two SUM queries per product).
  But for a warehouse with reasonable transaction volumes, this is fast.
```

**Inside GetInventoryAsync():**
```
GetInventoryAsync(string? search)
  │
  ├─► EnsureManager()
  │    └─ Only managers/admins can see inventory
  │
  ├─► factory.CreateDbContextAsync()
  │
  ├─► Build the query:
  │    IQueryable<Object> query = db.Objects
  │      .AsNoTracking()             ◄── Read-only, no change tracking
  │      .Include(x => x.Unit)
  │      .Include(x => x.Supplier)  ◄── Load related tables for display
  │
  ├─► If search is provided:
  │    query = query.Where(x =>
  │      x.Id.Contains(search) ||
  │      (x.DisplayName ?? "").Contains(search)
  │    )
  │    └─ Filter: product ID or name contains the search text (case-sensitive)
  │
  ├─► PROJECTION to avoid loading unnecessary data:
  │    var data = await query.Select(x => new
  │    {
  │      x.Id,
  │      x.DisplayName,
  │      Unit     = x.Unit.DisplayName ?? "",
  │      Supplier = x.Supplier.DisplayName ?? "",
  │      Input    = x.InputInfos.Sum(i => (int?)i.Count) ?? 0,
  │      Output   = x.OutputInfos.Sum(o => (int?)o.Count) ?? 0
  │    })
  │    .OrderBy(x => x.DisplayName)
  │    .ToListAsync()
  │    │

  │         SELECT o.Id, o.DisplayName, u.DisplayName AS Unit,
  │                s.DisplayName AS Supplier,
  │                (SELECT SUM(Count) FROM InputInfo WHERE ObjectId=o.Id) AS Input,
  │                (SELECT SUM(Count) FROM OutputInfo WHERE ObjectId=o.Id) AS Output
  │         FROM Object o
  │         LEFT JOIN Unit u ON o.UnitId = u.UnitId
  │         LEFT JOIN Supplier s ON o.SupplierId = s.SupplierId
  │         WHERE (o.Id LIKE '%search%' OR o.DisplayName LIKE '%search%')
  │         ORDER BY o.DisplayName
  │
  │       Using .Select() projection means EF only fetches these 7 columns,
  │       not every column from all three tables. This is efficient.
  │
  └─► Map to InventoryRow DTOs:
       return data.Select(x => new InventoryRow(
         x.Id,
         x.DisplayName ?? "",
         x.Unit,
         x.Supplier,
         x.Input,
         x.Output,
         x.Input - x.Output         ◄── Stock computed in memory (simple subtraction)
       )).ToList()

Back in ViewModel:
       Replace(Rows, inventoryRows)
       └─► DataGrid with AutoGenerateColumns="True"
            automatically shows columns: Id, Product, Unit, Supplier, Input, Output, Stock
```

## 9. Architecture Map

### Complete call chain for a user action

```
USER CLICK
  │
  ▼
[View XAML]                         ◄── Button Command="{Binding XxxCommand}"
  │  WPF binding system finds the ICommand on DataContext
  ▼
[ViewModel]                         ◄── [RelayCommand] private Task XxxAsync() => ...
  │  CommunityToolkit-generated ICommand wrapper
  │  Calls ExecuteAsync() — ensures IsBusy guard, error handling
  ▼
[WarehouseService]                  ◄── Singleton, holds currentUser session
  │  EnsureManager() / EnsureAdmin() / EnsureAuthenticated()
  │  ValidationHelper checks
  │
  ├─► [WarehouseDbContext]          ◄── Fresh instance per operation (factory pattern)
  │     AsNoTracking() for reads
  │     SaveChangesAsync() for writes
  │     Transactions for critical operations
  │
  ▼
[SQL Server]                        ◄── RalseiWarehouse database
  │  Tables: User, Role, Object, Unit, Supplier, Customer,
  │          Input, InputInfo, Output, OutputInfo
  │
  └─► Result flows back up:
       Service → ViewModel.Replace(collection, result) → DataGrid updates

ERROR PATH:
  SQL Server error
    → DbUpdateException
      → Service catches, re-throws as InvalidOperationException with user message
        → ExecuteAsync catches Exception
          → ViewModelBase.Message = exception.Message
            → TextBlock in View shows red error text
```

### Authorization Flow

```
Every service method call:
  │
  ├─► EnsureAuthenticated()
  │    └─ currentUser != null → OK
  │    └─ currentUser == null → throw UnauthorizedAccessException("Login is required.")
  │
  ├─► EnsureManager()
  │    └─ EnsureAuthenticated() first
  │    └─ currentUser.Role.DisplayName contains "Manager" or "Admin" → OK
  │    └─ else → throw UnauthorizedAccessException("Manager permission is required.")
  │
  └─► EnsureAdministrator()
       └─ EnsureAuthenticated() first
       └─ currentUser.Role.DisplayName contains "Admin" → OK
       └─ else → throw UnauthorizedAccessException("Administrator permission is required.")

Which methods use which guard:
  - LoginAsync:          NO guard (you're not logged in yet)
  - CreateInputAsync:    EnsureAuthenticated() only
  - CreateOutputAsync:   EnsureAuthenticated() only
  - GetRecent*:          NO guard (all logged-in users see history)
  - GetUnits/Suppliers/Customers/Products (CRUD):  EnsureManager()
  - GetUnits/Suppliers/Customers/Products (GET):    called by Save methods which already guard
  - GetInventoryAsync:   EnsureManager()
  - GetRolesAsync:       EnsureAdministrator()
  - SaveRole/CreateUser: EnsureAdministrator()
```

### The Password Flow

```
Creating a new user:
  Admin types password → SaveUserAsync
    → ValidationHelper.Require(password, "Password")
    → SecurityHelper.HashPassword(password)
      → RandomNumberGenerator.GetBytes(16)        ◄── 16-byte random salt
      → Rfc2898DeriveBytes.Pbkdf2(
          password, salt, 210000, SHA256, 32)      ◄── 210K iterations, 32-byte hash
      → return "PBKDF2-SHA256$210000$base64salt$base64hash"
    → Stored in User.Password column

Stored format: "PBKDF2-SHA256$210000$JC8dGtR3...(salt)...$Ab3F...(hash)..."
                      ^           ^       ^                 ^
                   algorithm  iterations  salt (Base64)     hash (Base64)

Logging in:
  User types password → LoginAsync
    → Check if stored starts with "PBKDF2-SHA256$"
      YES → SecurityHelper.VerifyPassword(password, stored)
        → Parse stored: extract algorithm, iterations, salt, expected hash
        → Re-compute: Pbkdf2(password, salt, iterations, SHA256, 32)
        → FixedTimeEquals(actual, expected)
        → Match → login succeeds
      NO (legacy plain text) →
        → Direct comparison: stored == password
        → Match → auto-upgrade: HashPassword(password), save to DB
        → Next login uses the hash path
```

## Summary: What happens when...

| User action | Execution chain |
|---|---|
| **Launch app** | `App.OnStartup` → Load config → Build DI → `ShowLogin()` → LoginWindow appears |
| **Log in** | `LoginCommand` → `LoginViewModel.LoginAsync` → `service.LoginAsync` → Query Users table → Verify password → `Succeeded` event → Create `MainViewModel` + `MainWindow` → Pre-load data |
| **Refresh master data** | `LoadCommand` → `MasterDataViewModel.LoadAsync` → 4 parallel service calls → `Replace()` all 4 DataGrids |
| **Save a supplier** | `SaveSupplierCommand` → `WriteAsync(service.SaveSupplierAsync)` → Validate → `SaveChangesAsync` → Reload all 4 lists → "Saved successfully." |
| **Delete a unit** | `DeleteUnitCommand` → `WriteAsync(service.DeleteUnitAsync)` → Find → Remove → `SaveChangesAsync` → If FK violation: "The unit is used by a product." |
| **Complete import** | `SaveInputCommand` → Map lines to `InputRequest` DTOs → `service.CreateInputAsync` → Validate lines → INSERT header + line items → Clear lines → Reload history |
| **Complete export** | `SaveOutputCommand` → `service.CreateOutputAsync` → Serializable transaction → For each line: SUM imports - SUM exports → Check stock → INSERT header + line items → Commit |
| **Search inventory** | `LoadCommand` → `service.GetInventoryAsync(search)` → Filter products → SUM subqueries for stock → Map to `InventoryRow` → Display in grid |
| **Save new user** | `SaveUserCommand` → `service.SaveUserAsync(user, password)` → Hash password → Check unique username → INSERT or UPDATE → Refresh roles+users list |
| **Close app** | `MainWindow` closes → `App.OnExit` → `provider.Dispose()` → cleanup |
